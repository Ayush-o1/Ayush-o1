## ✦ &nbsp;&nbsp; The Problems I Solved &nbsp;&nbsp; ✦

*These aren't projects. They're answers to real engineering questions.*

</div>

<br>

<details open>
<summary><strong>&nbsp;⟁ &nbsp; How do you make any app provider-agnostic for LLMs, cheaper, and more reliable?</strong></summary>

<br>

> ### [ContextForge](https://github.com/Ayush-o1/contextforge)
>
> You build a proxy gateway that sits between your app and any LLM provider. ContextForge is OpenAI-API-compatible, so any existing app can use it with a one-line change. It adds semantic caching (so you never pay twice for the same query), multi-provider routing with fallback/retry logic (so you're never dependent on one provider), and context compression that cuts token overhead before the request even reaches the model.
>
> Deployed on Railway. Full CI/CD. This is not a toy — it's designed as a production drop-in.
>
> `FastAPI` · `Redis` · `Docker` · `Railway` · `GitHub Actions`

<br>

</details>

<details open>
<summary><strong>&nbsp;⟁ &nbsp; How do you build a backend where every actor can only do exactly what they're allowed to?</strong></summary>

<br>

> ### [FreightFlow](https://github.com/Ayush-o1/FreightFlow)
>
> You design RBAC from the ground up, not as an afterthought. FreightFlow is a logistics platform backend with JWT authentication and role-based access control across three distinct actor types — shipper, driver, admin. Permission boundaries are enforced at the middleware layer, not sprinkled through business logic. Clean route/controller/model architecture. Built to the standard I'd want in a real production codebase.
>
> `Node.js` · `Express` · `MongoDB` · `JWT`

<br>

</details>

<details open>
<summary><strong>&nbsp;⟁ &nbsp; How do you make prompt engineering reproducible, testable, and safe to iterate on?</strong></summary>

<br>

> ### [AutoPrompt](https://github.com/Ayush-o1/autoprompt)
>
> You treat prompts like code. Version them. Test them. Automate the evaluation. AutoPrompt uses YAML-driven template versioning so prompt changes are auditable and reviewable like any other diff. Automated scoring pipelines run on every commit via GitHub Actions. Integrates Gemini API. Every evaluation is reproducible.
>
> `FastAPI` · `Docker` · `GitHub Actions` · `Gemini API`

<br>

</details>

<details open>
<summary><strong>&nbsp;⟁ &nbsp; How do you eliminate the "clone and configure" contributor onboarding problem?</strong></summary>

<br>

> ### [Onboarder](https://github.com/Ayush-o1/Onboarder)
>
> You automate it entirely. Paste a GitHub URL. Get a working Docker environment in minutes. Zero manual configuration. Zero "follow these 15 steps." Onboarder handles environment generation for MERN stack projects. The experience should be: clone, run one command, start contributing.
>
> `Node.js` · `React` · `Docker`

## ✦ &nbsp;&nbsp; My Engineering Philosophy &nbsp;&nbsp; ✦

</div>

<br>

<table>
<tr>
<td width="50%" valign="top">

**What I've learned to love** ✓

- Systems with observable behavior you can trace
- APIs that do exactly what their contract says
- Error handling that's a first-class concern, not an afterthought
- Code that a new engineer can read without a walkthrough
- Boring, proven technology where it works
- Deployments that tell you what they're doing

</td>
<td width="50%" valign="top">

**What I've learned to hate** ✗

- Magic in the critical path
- Auth implemented as a thought-exercise at the end
- "Just read the code" as documentation
- Abstractions that make the simple complex
- Frameworks chosen to be impressive, not effective
- Observability added retroactively

</td>
